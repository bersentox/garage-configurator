# site_body

Технический документ для автономного блока `site_body` в `blocks/site-body/`.

## Роль блока

`site_body` больше не повторяет hero и configurator. Блок начинается сразу после configurator и работает как **decision-acceleration layer**:

- продолжает инерцию после первичного выбора;
- быстро показывает контраст между хаосом и системой;
- переводит пользователя к следующему действию без длинных объяснений;
- снимает остаточные сомнения через FAQ и trust layer.

## Новая архитектура

Корневой контейнер остаётся тем же:

- `<section id="site-body" class="sb">`

Внутри размещаются 9 последовательных секций:

1. **Intro transition**
   - короткий мост после configurator;
   - заголовок + одна строка контекста;
   - без длинного вступления.
2. **Comparison Row 1**
   - тема: `Где теряется контроль | Где он фиксируется`;
   - две симметричные колонки `хаос ↔ система`;
   - короткие парные пункты без табличного вида.
3. **Comparison Row 2**
   - тема: `Где возникают проблемы качества | Как это решено`;
   - две симметричные колонки со скрытыми рисками и заранее решёнными точками.
4. **Micro CTA**
   - короткий переход после контраста;
   - заголовок, одна строка пояснения и один primary CTA.
5. **System Navigator**
   - маршрут от первого контакта до готового результата;
   - 5–7 коротких шагов в формате карты / timeline.
6. **Action Navigator**
   - 3–5 коротких шагов `Вы / Мы`;
   - снимает трение перед действием;
   - содержит основной CTA.
7. **FAQ**
   - 6–10 релевантных вопросов по выбору, срокам, основанию, монтажу, гарантии, логистике и кастомизации;
   - формат disclosure / accordion;
   - короткие ответы без водянистых блоков текста.
8. **Trust Layer**
   - аккуратные подблоки: подход, достижения, отзывы, проекты;
   - сдержанный premium-tone без крикливых цифр и дешёвого лендинга.
9. **Footer**
   - лаконичное завершение блока;
   - финальный переход к следующему шагу.

## HTML-контракт

- Использовать семантические секции, header, article, footer, ol, ul, details, summary, a.
- Не возвращать hero/configurator внутрь `site_body`.
- Не смешивать новую структуру со старой историей, старыми фазами и legacy-контентом.
- Все элементы остаются внутри namespace `.sb`.

## CSS-принципы

- Изоляция только через `.sb` namespace.
- Mobile-first адаптив обязателен.
- Основа визуального языка:
  - высокий контраст и чёткая иерархия;
  - короткие тезисы и много воздуха;
  - сильные comparison rows без Excel-таблиц и без карточного лендинга;
  - токены для цвета, типографики, spacing, radius и target size.
- Избегать случайных значений и декоративного шума.

## JS-принципы

- Только минимально необходимый JS.
- Допустим только лёгкий enhancement для disclosure / FAQ.
- Без библиотек, без тяжёлого интерактива, без зависимости от внешних блоков.
- Логика должна работать только внутри `#site-body`.

## Доступность

- `focus-visible` для интерактивов.
- Кликабельные зоны не меньше target token.
- Если disclosure управляется JS, синхронизировать `aria-expanded`.
- Сохранять читаемый контраст и keyboard accessibility на нативных элементах.

## Ограничения разработки

Разрешено менять только:

- `site-body.html`
- `site-body.css`
- `site-body.js`
- `README.md`

Нельзя:

- создавать новые файлы;
- менять структуру директории `blocks/site-body/`;
- трогать hero, configurator, root index или другие части репозитория.
